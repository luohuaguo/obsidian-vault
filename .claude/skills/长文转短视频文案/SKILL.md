# 长文转短视频文案

> 公众号长文 → 保留骨架 → 逐层压缩 → 500-1000字短视频文案

## 功能概述

将公众号长文案整体压缩为短视频文案，**保留原文的结构骨架**，通过删减展开、合并段落、压缩描述来瘦身。

核心原则：**这是瘦身，不是重写。**

```
公众号长文 → 识别骨架 → 标记可砍区域 → 逐层压缩 → 自检 → 存档
```

## 输入参数

| 参数 | 必填 | 说明 | 默认值 |
|------|------|------|--------|
| 源文案 | 是 | 公众号文章路径或全文 | - |
| 目标字数 | 否 | 500/700/1000 | 700 |
| 平台 | 否 | 视频号/抖音/小红书 | 视频号 |

## 铁律（不可违反）

### 铁律一：骨架保留

原文的章节结构（##标题）是骨架，压缩时保留整体叙事线，允许：
- 合并相邻章节
- 删除整个章节（如果不影响主线）
- 章节内部大幅删减

不允许：
- 改变叙事顺序
- 添加原文没有的内容
- 改变原文的核心观点

### 铁律二：生成即存档

文案生成后第一件事是立即Write存档，不等确认。

## 完整工作流程

### Step 1：分析原文骨架

读取源文案，识别：
1. 总字数
2. 章节结构（每个##标题）
3. 每个章节的核心信息（一句话概括）
4. 结尾引流段落（标记为保留区）

输出骨架分析：
```
原文：{字数}字，{N}个章节
目标：压缩到{目标字数}字，需砍{X}字

骨架：
- {章节1标题}（{字数}字）← 核心：{一句话}
- {章节2标题}（{字数}字）← 核心：{一句话}
- ...
- 结尾引流（{字数}字）← 需压缩重写（短视频引流要更短更直接）

压缩方案：
- 砍掉：{章节X}（原因：不影响主线）
- 大幅压缩：{章节Y}（原因：展开太多，核心一句话就够）
- 合并：{章节A+B}（原因：可以合成一段）
- 保留：{章节Z}（原因：差异化内容/核心价值）

要不要按这个方案压？
```

**等待用户确认后再压缩。** 用户可能会说"这个章节要留"或"那个可以全砍"。

---

### Step 2：逐层压缩

按用户确认的方案执行压缩。

#### 压缩手法优先级（从最不伤内容到最伤内容）

1. **删冗余铺垫**：每个章节开头的过渡句、背景解释，直接砍
2. **合并展开项**：原文分1/2/3步展开的，合并成一段话
3. **压缩描述**：3句话说的事压成1句
4. **删整个章节**：对主线贡献最小的章节整段删
5. **删图片**：如果图片不是核心理解必需的，删掉（减少视觉噪音）
6. **压缩引流段**：短视频引流要比公众号更短更直接。原文引流段不能直接搬，需要重写压缩：砍掉重复定位描述，一句话说清产品+价格+方向，控制在2-3句以内

#### 压缩时的语言规则

- 保持原文的语气和用词习惯，不要改写
- 压缩是删减，不是换一种说法重新表达
- 如果原文用了"说白了就是"，压缩版也用，不要换成"本质上是"
- 保留原文的金句和有力表达，砍的是水分不是肉

#### 口播断句规则（压缩时必做，不要等用户提）

- 每句话不超过25个字，超过必须断开
- 一句话里逗号超过3个，必须在合适位置改成句号或分段
- 断句方式：逗号→句号、长句拆成两段、删掉连接词让句子独立
- 断句后默读一遍，确保每句都能一口气说完

#### 短视频口播适配规则（压缩时必做）

- **阿拉伯数字转中文**：口播稿中所有数字用中文表达。100→一百，2026→二零二六，1万→一万。唯一例外：产品名中的数字保持原样（如GPT-5.4）
- **载体词替换**：原文中的"这篇文章"统一改为"这条视频"或"这个视频"，因为输出载体是短视频不是图文
- **去掉双引号**：自动去掉所有双引号，改用直接表述
- **禁止分隔线**：正文中不使用 `---` 水平分隔线，章节之间用空行分隔即可

#### 配图处理

**优先复用原文图片**：源公众号文章已经配好的图片，压缩时默认全部保留并搬到对应位置，不需要重新判断删不删。用户已经花时间选图配图了，短视频版直接复用是最高效的做法。

仅在以下情况删图：
- 用户主动说"图片太多了"或"不要图"
- 压缩后某个章节被整段删除，该章节的图片随之删除

**配图时机**：压缩时同步处理（和断句一样，不要等用户提）。从原文对应位置把图片标记搬过来。

---

### Step 3：存档

**生成完成后立即存档，不等确认。**

文件名格式：先用源公众号文案同名暂存。等Step 4标题生成后，用户选定短视频标题时，重命名文件为 `{日期}-{短视频标题}.md`。

Frontmatter格式：
```yaml
---
title: {原文标题}-短视频版
date: {日期}
platform: {平台}
type: 短视频文案
status: 草稿
source: {源文案路径}
original_word_count: {原文字数}
compressed_word_count: {压缩后字数}
compression_ratio: {压缩比例}
tags:
  - 短视频
  - 长文压缩
  - {相关标签}
---
```

---

### Step 4：短视频标题生成

压缩文案存档后，基于压缩后的短视频文案重新生成标题。

**为什么要重新生成**：公众号标题和短视频标题风格不同。公众号标题可以长、可以悬念；短视频标题要更短、更口语、更直接，适配竖屏封面和前3秒注意力。

**执行方式**：调用「标题生成」skill，输入压缩后的短视频文案，生成5-10个短视频标题候选。

**用户选定后**：
1. 将选定标题更新到文件的frontmatter `title` 字段
2. 重命名文件为 `{日期}-{短视频标题}.md`
3. 告知用户文件已重命名

**等待用户选定后再重命名，不要自动选。**

---

### Step 5：自检

压缩后逐项检查：

```
□ 骨架完整：主线叙事是否通顺？删掉的部分是否影响理解？
□ 字数达标：是否在目标字数范围内？
□ 风格一致：语气、用词是否与原文保持一致？
□ 引流保留：结尾引流段是否保留？
□ 独立完整：不看原文，这篇能独立看懂吗？
□ AI味检查：压缩过程中是否引入了AI风格的表达？
□ 数字中文化：所有阿拉伯数字是否已转为中文？（产品名除外）
□ 载体词替换：这篇文章是否已改为这条视频？
□ 双引号清除：正文中是否还有残留的双引号？（必须全部去掉）
□ 配图复用：原文图片是否已搬到对应位置？
```

不通过的项目直接修改，不需要用户参与。

---

### Step 6：展示压缩结果

```
压缩完成：{原文字数}字 → {压缩后字数}字（压缩{比例}%）

砍掉的内容：
- {删掉了什么，为什么}

保留的核心：
- {保留了什么，为什么}

文案已存档到：{文件路径}

要不要再调整？可以说"再短一点"或"这段要保留"。
```

## 迭代逻辑

用户说"再短一点"时：
1. 在当前版本基础上继续压缩，不从原文重新开始
2. 用Edit直接改已存档的文件，不重新生成
3. 每次压缩都告诉用户砍了什么

用户说"这段要保留"时：
1. 标记该段为保护区
2. 从其他地方找压缩空间

用户说"开头改一下"时：
1. 确认改成什么风格（直述/其他）
2. 只改开头，其他不动

## 快速启动

用户可以通过以下方式启动：

1. **指定文章**：
```
帮我把这篇压短：{文章路径}
```

2. **指定字数**：
```
这篇太长了，压到800字
```

3. **快速模式**：
```
转成短视频文案
```
