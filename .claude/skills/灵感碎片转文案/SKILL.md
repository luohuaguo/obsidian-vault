# 灵感碎片转文案

> 你负责想，AI负责判断怎么写，然后串联整条生产线

## 🎯 功能概述

这是一个**智能路由 + 流水线调度 skill**，解决两个核心问题：
1. 你脑子里有个想法，但不知道该用什么类型、什么框架来写
2. 文案写完后，标题、排版、核查、存档等下游环节容易遗漏

```
碎片文字输入 → 内容类型判断 → 确认 → 调用对应文案 skill → 标题生成 → 排版优化 → 事实核查+敏感词检测 → 存档+反馈闭环
```

| `执行模式` | ❌ 否 | 自动模式/交互模式 | **询问用户选择** |
| `目标平台` | ❌ 否 | 公众号/小红书/视频号 | 公众号 |

## 🔄 完整工作流程

### Step 1：碎片解析 + 类型判断

接收用户输入的碎片文字后，执行以下分析：

**1.1 提取核心信息**

从碎片文字中提取：
- 核心主题：这段文字在讲什么？（一句话概括）
- 关键人物：涉及谁？（作者自己、他人、公众人物）
- 关键数据：有没有具体数字？（金额、时间、数量等）
- 关键观点：有没有明确的立场或判断？
- 关键方法：有没有可复制的步骤或方法论？
- 关键故事：有没有完整的故事线？（起因→经过→结果）
- 情绪基调：这段文字的情绪是什么？（震撼、共鸣、焦虑、期待等）

**1.2 内容类型判断**

根据提取的信息，按以下决策树判断：

```
碎片文字中有客户反馈/学员案例/对话截图/成交故事？
  ├─ 是 → 「客户见证」
  │    └─ 目标：私聊咨询 / 加社群 / 购买课程 / 领取资料
  └─ 否 → 继续判断
       │
       碎片文字中有可复制的方法/步骤/教程？
         ├─ 是 → 「教知识」
         │    └─ 细分：方法论拆解 / 追求更好 / 解决问题 / 降低成本 / 丰富谈资
         └─ 否 → 继续判断
              │
              碎片文字中有明确的观点/立场/判断？
                ├─ 是 → 「聊观点」
                │    └─ 细分：热点借势 / 认知颠覆 / 行业洞察 / AI速评
                └─ 否 → 继续判断
                     │
                     碎片文字中有完整的故事线（人物+事件+结果）？
                       ├─ 是 → 「讲故事」
                       │    └─ 细分：小有成就型 / 成功案例型 / 平凡英雄型 / 猎奇苦难型 / 共鸣苦难型
                       └─ 否 → 继续判断
                            │
                            碎片文字中有过程展示（做了什么+怎么做的+结果）？
                              ├─ 是 → 「晒过程」
                              └─ 否 → 默认「聊观点」（大部分碎片想法本质是观点）
```

**特殊情况处理**：
- 如果碎片文字同时符合多个类型（比如既有方法论又有故事），列出所有可能的类型，按匹配度排序，推荐最优的那个
- 如果碎片文字信息量太少（不足50字），提示用户补充更多细节

**1.3 输出判断结果**

```markdown
## 🧠 碎片解析

### 核心主题
{一句话概括}

### 提取的关键信息
- 关键人物：{人物}
- 关键数据：{数据}
- 关键观点：{观点}
- 关键方法：{方法}
- 关键故事：{故事线}
- 情绪基调：{情绪}

### 内容类型判断
**推荐类型**：{教知识/聊观点/讲故事/晒过程}
**细分方向**：{具体细分}
**判断理由**：{为什么是这个类型，50字以内}

### 其他可选类型（如有）
- {类型2}：{理由}
- {类型3}：{理由}
```

## 🛑 检查点1：用户确认类型 + 选择模式

**必须暂停**，等待用户确认。

**提示语**：
```
我判断这段内容适合写成「{类型}」类文案，理由是{理由}。

请选择创作模式：

🚀 自动模式：我直接跑完整条流水线（文案→标题→排版→核查→存档），中间只在标题环节让你选择。适合素材充足、想快速出稿的情况。

🔧 交互模式：一步步来，每一步都等你确认（选框架→选选题→补素材→写文案→自检）。适合重要选题、想深度打磨的情况。

回复示例：
- "自动" 或 "快速出稿" → 自动模式
- "交互" 或 "一步步来" → 交互模式
- 也可以同时换类型，比如："交互，换聊观点"
```

### 模式说明

**🚀 自动模式**：
- 本 skill 直接读取所有知识库，自主完成框架选择、文案生成、标题生成等全流程
- 仅在「标题选择」环节暂停等待用户确认
- 适合：素材充足、赶时间、不需要逐步打磨

**🔧 交互模式**：
- 调用对应的下游文案 skill（教知识文案/聊观点文案等），由该 skill 按自身的完整交互流程执行
- 下游 skill 的每一步都会等待用户确认后再进行下一步
- 下游 skill 完成后，流程回到本 skill 继续执行标题生成、排版、核查、存档
- 适合：重要选题、素材不足需要补充、想锻炼表达能力

### Step 2：调用对应文案 skill（含知识库注入）

用户确认类型和模式后，根据模式执行不同流程：

#### 🚀 自动模式下的 Step 2

本 skill 自主完成文案生成，不调用下游 skill 的交互流程：

1. 读取所有知识库（框架说明书、风格萃取、个人自传、社群总结、AI创作规范、开篇模板库、引流钩子模板）
2. 自主选择最优框架（基于框架使用说明书和框架效果数据）
3. 直接生成完整文案
4. 生成完成后，继续执行 Step 3-6

**不暂停**，一气呵成完成文案生成。

#### 🔧 交互模式下的 Step 2

调用对应的下游文案 skill，由该 skill 按自身完整交互流程执行：

**路由表**：

| 内容类型 | 调用 skill | 说明 |
|---------|-----------|------|
| 客户见证 | 「客户见证转化文案」skill | 碎片文字作为客户素材直接传入，由该 skill 完成画像提取、四段式生成、转化钩子设计等全流程 |
| 教知识 | 「教知识文案」skill | 碎片文字作为素材直接传入，由该 skill 完成框架匹配、文案生成等全流程 |
| 聊观点 | 「聊观点文案」skill | 碎片文字作为素材直接传入，由该 skill 完成框架匹配、文案生成等全流程 |
| 讲故事 | 「讲故事文案」skill | 碎片文字作为素材直接传入，由该 skill 完成故事类型匹配、文案生成等全流程 |
| 晒过程 | 「实战复盘转内容」skill | 碎片文字作为素材直接传入 |

**调用时的素材传递**：

调用对应 skill 时，必须将以下信息一并传入：
1. 用户的原始碎片文字（完整保留，不做删改）
2. Step 1 提取的核心信息（主题、人物、数据、观点、方法、故事线）
3. 确认的内容类型和细分方向

**⚠️ 知识库注入要求**：

无论调用哪个下游 skill，在文案生成阶段必须读取以下知识库：

1. **框架匹配**：读取`框架使用说明书`和`框架效果数据`，选择最优框架
2. **风格注入**：读取`公众号文案风格萃取`和`卡兹克文案规律萃取`
3. **信任建设**：读取`个人自传`，提取真实经历片段注入文案
4. **观点升华**：读取`社群总结`，注入专业观点和独家方法论
5. **去AI味**：读取`AI创作规范`，按70%+30%原则检查
6. **开篇优化**：按内容类型读取对应的`开篇模板库`
7. **结尾引流**：读取`公众号结尾引流钩子模板`，匹配引流方式

**调用后**：下游 skill 完成文案生成后，流程回到本 skill 继续执行 Step 3-6。

### Step 3：标题生成

文案完成后，读取`卡兹克标题公式萃取`，执行标题生成：

1. 从9大标题公式中选3-4个适合当前选题的公式
2. 从S级词汇库中选择对应词汇填入
3. 生成10个备选标题
4. 检查是否符合5个核心规律
5. 排除禁用句式
6. 给出TOP 3推荐

**🛑 检查点2：用户确认标题**

必须暂停，等待用户选择标题。

---

### Step 4：排版优化

标题确认后，调用「公众号排版」skill 进行排版优化。

### Step 5：事实核查 + 敏感词检测

**5.1 事实核查**

检查文案中的：
- 具体数据是否来自用户提供的真实素材
- 案例是否真实（标注"需验证"的部分）
- 方法是否可行

**5.2 敏感词检测**

调用「敏感词检测」skill，覆盖公众号平台规则，给出替换建议。

---

### Step 6：存档 + 反馈闭环

**6.1 更新框架效果数据**

在`框架效果数据.md`中记录本次使用的框架：
- 框架名称
- 使用日期
- 文章标题
- 使用次数+1

**6.2 文案存档**

> ⚠️ **铁律：文案正文生成后必须立即存档草稿，不得等定稿。** 上下文窗口有限，对话结束后内容永久丢失。后续所有修改直接 Edit 该文件。

**存档规范**：文案保存到 `07.发布文案/2026年/草稿/` 目录，文件名格式 `YYYY年M月D日-标题关键词.md`。frontmatter 必须遵循 `00.系统配置/frontmatter规范.md` 中的统一规范，必填字段：title、type、status（初始为"草稿"）、created、tags、source_type。**存档时机：文案正文输出完毕后立即执行，不等标题确认，不等自检通过。**

告诉用户："✅ 文案已保存到 `07.发布文案/2026年/草稿/` 目录，在Obsidian左侧栏确认一下。如果没看到，告诉我，我手动帮你存"

**6.3 文案状态更新**

将文案 frontmatter 中的`status`从`草稿`改为`待发布`。

## ⚠️ 重要规则

### 必须遵守

1. **不改原文**：碎片文字原封不动传给下游 skill，不做删改或"优化"
2. **必须暂停确认**：类型判断+模式选择（检查点1）和标题选择（检查点2）必须等用户确认
3. **模式必须明确**：用户必须选择自动模式或交互模式后才能进入 Step 2，不可默认跳过
4. **信息不足要追问**：如果碎片文字不足50字或关键信息缺失，先追问再判断
5. **多类型要说明**：如果同时符合多个类型，必须列出所有可能并推荐最优
6. **知识库必读**：Step 2 的7项知识库注入不可跳过（无论哪种模式）
7. **下游环节不可省略**：标题生成、排版、核查、存档必须执行，不能在文案生成后就结束

### 禁止行为

1. ❌ 禁止跳过类型判断直接生成文案
2. ❌ 禁止在用户未确认类型时自动调用下游 skill
3. ❌ 禁止修改用户的原始碎片文字
4. ❌ 禁止在信息明显不足时强行判断
5. ❌ 禁止文案生成后跳过标题生成、排版、核查等下游环节
